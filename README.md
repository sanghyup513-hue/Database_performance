# 프로젝트: PostgreSQL vs Altibase 성능 비교 벤치마크

## 목적
PostgreSQL과 Altibase의 성능을 테스트 유형별로 비교하고, 결과를 자동으로 수집·리포트하는 작업을 돕는다. 최종 리포트는 고객에게 제공된다.

## 확정된 아키텍처 (전제로 삼을 것)
- 측정 대상 두 DB는 K8s 클러스터 밖 **외부 서버**에 각각 위치한다. 서버 성능은 넉넉하여 대용량 규모의 쿼리·데이터셋도 처리 가능하다.
- 부하 생성·모니터링·자동화는 **K8s 클러스터**에서 수행한다. (DB는 K8s에 없음)
- 부하 도구는 **BenchBase**(JDBC 기반). PostgreSQL·Altibase 둘 다 JDBC로 연결하며, 트랜잭션 유형별 지연시간·처리량 로그를 수집한다.
- 지표 수집은 Prometheus가 외부 DB 서버의 exporter를 원격 scrape, 시각화는 Grafana.
- 자동화는 K8s Job/CronJob으로 워크로드 실행 → 유형별 결과 수집 → 집계 스크립트(Python/pandas)로 비교 리포트.
- **Chaos Mesh는 사용하지 않는다.** 장애 주입 테스트는 이 프로젝트 범위 밖이다. (이미 검토 후 제외 결정)

## 도구 제약 (틀리기 쉬운 부분)
- sysbench, HammerDB, pgbench는 **Altibase를 지원하지 않는다.** 두 DB 공통 비교에는 쓰지 말 것. 공통 하니스는 JDBC 기반(BenchBase, 또는 Altibase JDBC 직접 호출 커스텀 하니스)이어야 한다.

## 테스트 환경 정의 (사전 고정 후 리포트에 반드시 명시)
- **DB 버전 고정**: PostgreSQL·Altibase 각각의 정확한 버전을 고정하고 리포트에 기재한다. 재현성과 신뢰성의 기본 요소.
- **데이터셋/스케일 정의**: 데이터 규모(행 수·용량), 스키마, 스케일 팩터(예: TPC-C warehouse 수)를 명시하고, 두 DB에 동일한 데이터로 적재한다. "대용량"의 구체 수치를 반드시 정의할 것.
- **합성 데이터 vs 실데이터** 여부를 명시한다.

## 테스트 유형
1. Read-only (SELECT: 포인트 조회, 범위 스캔, 조인)
2. Write-heavy (UPDATE / INSERT)
3. Mixed OLTP (TPC-C 류)
4. 고동시성 확장성 (커넥션/스레드 단계적 증가)
5. 대용량 워크로드 (대용량 스캔·집계·조인(OLAP/TPC-H 류), 대량 배치 UPDATE/DELETE)

측정 지표: TPS·QPS, 지연시간 p50/p95/p99, 에러율.

## 비교 공정성 원칙 (항상 지킬 것)
- **튜닝 기준선**: 두 DB를 기본 설정으로 둘지, 각자 권장 튜닝을 적용할지 사전에 결정하고 리포트에 명시한다. (※ 미결정 — 정해지면 이 항목을 확정 기재할 것. 결정 전에는 클로드가 임의로 한쪽만 튜닝하지 않는다.)
- 부하 생성기와 두 DB 서버 간 **네트워크 경로/RTT 동일** (측정 전 ping/mtr 확인, 리포트에 기록).
- 워밍업(ramp-up) 후 측정, 각 측정은 3회 이상 반복하여 중앙값 사용.
- **결과 신뢰성**: 반복 측정 간 편차가 크면(예: 변동계수 과다) 원인 확인 후 재측정한다. 명백한 이상치는 기준을 정해 제외하고, 제외 사실을 리포트에 남긴다.
- Altibase는 인메모리 성격이 강하므로 **데이터셋 메모리 이내/초과 시나리오를 둘 다** 측정.
- 부하 생성기(클라이언트) 자원도 모니터링 — 클라이언트 병목과 DB 성능을 혼동하지 말 것.

## 리포트 산출물 (고객 제공용 — 무수정 납품 품질)
- 최종 산출물은 **고객에게 그대로 전달 가능한 완성도**여야 한다. 사용자가 추가로 디자인을 손볼 필요가 없는 수준으로 만든다.
- 기본 포맷은 **PDF**. (요청 시 자체완결형 HTML도 가능)
- 단순 마크다운 표 나열 금지. 다음을 갖춘 전문적 레이아웃으로 작성한다:
  - 표지 / 제목 / 작성일 / 테스트 환경 요약
  - 핵심 요약(Executive Summary)을 맨 앞에 배치 — 결론을 한눈에
  - 원시 수치 표보다 **차트 위주**로 시각화 (테스트 유형별 처리량·지연시간 비교)
  - 두 DB를 **일관된 색상 코드**로 구분 (전체 문서에서 동일 규칙 유지)
  - 섹션 구조: 요약 → 테스트 환경/방법 → 유형별 결과 → 종합 분석 → 부록(상세 수치)
- 글꼴·여백·정렬·차트 스타일이 일관되고 인쇄에 깨지지 않게 한다.
- 수치 해석은 데이터에 근거하여 중립적으로 서술. 결과를 임의로 한쪽에 유리하게 윤색하지 않는다.

## 작업 환경 / 구현 메모 (진행 중)
- **테스트 클러스터 ≠ 운영 대상 서버**: 현재 매니페스트 검증은 단일 노드 테스트 K8s 클러스터에서 수행한다. 부하 도구를 실제 설치할 운영 대상 서버는 별도이며, 모든 구성은 운영 서버 제약을 전제로 설계한다. (접속 자격증명은 코드/문서에 평문으로 넣지 않음 — 아래 보안 항목 참고)
- **운영 서버 기존 모니터링 스택 보호**: 운영 대상 서버에는 이미 `mstr-prometheus` 네임스페이스에 kube-prometheus-stack(prometheus-operator, kube-state-metrics, node-exporter, Prometheus 인스턴스)이 운영 중이다. **이 기존 리소스에 절대 영향을 주지 않는다.**
- **Prometheus 설치 방식 — Standalone (확정)**: 벤치마크용 Prometheus는 **우리 전용 네임스페이스 안에 독립 인스턴스**(Deployment/StatefulSet + ConfigMap scrape 설정)로 격리 설치한다. 기존 prometheus-operator/cluster-scoped CRD를 재사용하거나 `kube-prometheus-stack`을 통째로 재설치하지 않는다(충돌 위험). 외부 DB exporter는 이 독립 Prometheus가 직접 원격 scrape.
- **K8s 리소스 위치**: 부하 생성기·워크로드 Job/CronJob·벤치마크용 Prometheus는 모두 전용 네임스페이스 `benchmark`에 배치한다. 매니페스트는 `k8s/` 디렉터리에 파일로 관리(재현 가능).

## 기밀 / 보안
- DB 접속 정보(호스트, 계정, 비밀번호, 포트 등 자격증명)를 지침·코드·리포트 등 어떤 산출물에도 평문으로 넣지 않는다. 환경변수·시크릿으로 분리한다.
- 고객 실데이터를 사용하는 경우 민감정보가 리포트나 로그에 노출되지 않도록 한다.
- 고객 제공 리포트에 내부용 메모·자격증명·미가공 접속 로그가 섞이지 않도록 검수한다.

## 응답 방식
- 답변 언어는 한국어.
- 위 확정 사항(외부 서버 구성, BenchBase, Chaos Mesh 제외, PDF 리포트)을 다시 제안하거나 뒤집지 말고 전제로 둘 것. 변경이 필요하면 명시적으로 확인받는다.
- Altibase 관련 사실(JDBC 설정, 포트, SQL 방언 등)은 불확실하면 추정하지 말고 확인하거나 검색한다.
- **Altibase 벤치마크 결과의 외부(고객) 공개**에 라이선스상 제약이 있을 수 있으므로, 공개 전 라이선스 조건을 확인할 것. (상용 DB는 공개 벤치마크를 제한하는 경우가 있음)
